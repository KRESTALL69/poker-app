# Project Context

## Что это

`Don't Worry Club` / `DWC` — это Telegram Mini App и web-приложение для покерного клуба и управления покерными турнирами.

Проект полностью self-hosted:

- GitHub repository
- собственный VPS (Docker, nginx, PostgreSQL)
- Telegram bot (webhook и Menu Button указывают на production-домен)
- Resend (отправка OTP-писем)

Supabase и Vercel **больше не используются** — полный переход на self-hosted
инфраструктуру завершён, см. `README.md` (актуальная архитектура) и
`docs/AUTH_MIGRATION.md`/`docs/POSTGRES_MIGRATION_ARCHITECTURE.md` (архив,
история перехода).

## Архитектура

- Next.js App Router
- React
- TypeScript
- Tailwind CSS
- PostgreSQL (self-hosted, Drizzle ORM)
- Собственная OTP-система + собственные сессии (без Supabase Auth)
- Локальное файловое хранилище для аватарок (без Supabase Storage)
- Telegram Mini App
- Telegram Bot API
- Google Sheets integration
- Docker + nginx (VPS)

Подробное описание — `README.md`, раздел «Архитектура».

Главный принцип архитектуры: бизнес-логика должна работать через внутреннюю сущность `player`, а не напрямую через Telegram.  
Telegram, email и web-вход — это разные способы получить текущего `player`.

## Основные сущности

- `players` — игроки, профиль, роль, Telegram-данные, email, ник, доступы
- `tournaments` — турниры, дата, лимит игроков, описание, локация, статус, цены
- `registrations` — записи на турниры, waitlist, отмены
- `results` — результаты турниров, место, нокауты, очки рейтинга
- `seasons` — сезоны
- `player_achievements` — прогресс достижений игрока
- `app_settings` — глобальные настройки приложения / feature flags
- Google Sheets — экспорт/импорт результатов и финансовый отчет

## Где что лежит

- `app/` — страницы и UI на App Router
- `app/api/` — API routes и webhook routes
- `features/` — бизнес-логика (обращается к данным только через Repository Layer, `lib/repositories/`)
- `lib/` — инфраструктурные helper-функции и интеграции
- `types/` — domain/database types
- `components/` — переиспользуемые UI-компоненты

## Авторизация

Сейчас есть несколько auth-сценариев.

### Telegram Mini App

Основной сценарий для пользователей внутри Telegram:

- приложение получает Telegram Mini App `initData`
- проверяет пользователя
- находит или создает `player` по `telegram_id`

### Web login через email (собственная OTP-система)

Web-версия приложения для пользователей, у которых не работает Telegram.

Flow:

- пользователь открывает `/login`
- вводит email
- `POST /api/auth/email/request-code` создаёт одноразовый код (таблица
  `email_otp_codes`) и отправляет его через Resend
- пользователь вводит код, `POST /api/auth/email/verify-code` проверяет его
  и ставит сессионную cookie `dwc_tg_session`
- приложение находит или создаёт `player` по email

Supabase Auth здесь больше не участвует — вся система своя, подробности
(TTL, лимиты попыток, хранение) в `docs/AUTH_MIGRATION.md`.

Важно:

- `email` хранится в `players`
- email нормализуется через `trim().toLowerCase()`
- в базе должен быть уникальный индекс по `lower(email)`
- если пользователь зашел через email, `telegram_id` может быть пустым
- бизнес-логика не должна падать без Telegram user

### Telegram web login

Вход через Telegram Login Widget для web-версии — **работает в проде**,
доведён до конца (в отличие от более раннего статуса "нестабильно"):
`/api/auth/telegram` инициирует OAuth-виджет, `/api/auth/telegram/callback`
проверяет подпись и ставит ту же сессионную cookie `dwc_tg_session`, что и
email-логин — то есть оба веб-сценария сходятся к одному механизму сессии.

Telegram Mini App auth работает отдельно (через `initData`, не через эту
cookie) и не завязан на неё.

## Email-привязка

Для Telegram-пользователей был реализован modal / bottom sheet с предложением привязать email:

- показывается только Telegram-пользователям
- только если у `player` нет email
- не блокирует приложение
- есть кнопка “Позже”
- после подтверждения email привязывается к текущему `player`

Сейчас показ этого modal управляется через глобальную настройку в админке.

## Поддержка через бота

Есть Telegram support через бота.

Текущая логика:

- `/start` показывает меню бота
- в меню есть кнопки:
  - “Открыть приложение”
  - “Поддержка”
- `/support` запускает support-сценарий
- deep link `?start=support` должен обрабатываться как support-сценарий
- Telegram визуально может показывать `/start`, это нормально
- бот должен различать обычный `/start` и `/start support`

Support-сценарий:

- бот пишет пользователю: “Напишите ваш вопрос, мы скоро ответим.”
- сообщение пользователя пересылается в админ-чат
- ответ админа реплаем возвращается пользователю

## Google Sheets

Реализовано:

- экспорт результатов турниров в Google Sheets
- импорт / синхронизация результатов из Google Sheets
- двусторонняя синхронизация Google Sheets ↔ приложение

- формулы на `Лист1` автоматически создаются при добавлении новой строки

Важно:

- аккуратно работать с кириллицей и UTF-8
- не ломать текущую структуру Google Sheets
- не менять финансовый отчет без необходимости

## Права и доступы

В `players` есть доступы:

- `can_access_free`
- `can_access_paid`
- `can_access_cash`

В админке есть раздел “Доступы игроков”.

Для отображения игрока в админке:

- если есть `telegram_id` — показывать Telegram ID
- если `telegram_id` нет, но есть `email` — показывать email
- если нет ни того, ни другого — показывать “Нет данных”

Поиск в админке должен учитывать:

- ник
- Telegram ID
- email

## Важные технические моменты

### Email

- email должен храниться в нижнем регистре
- перед записью использовать `trim().toLowerCase()`
- не допускать дублей по email без учета регистра
- если email уже занят другим `player`, показывать понятную ошибку
- resend-cooldown (60 сек) и лимит попыток (5) — на уровне собственной OTP-системы, не стороннего провайдера; см. `docs/AUTH_MIGRATION.md`

### Email OTP (своя система, без Supabase)

- web email login использует собственную OTP-систему (таблица `email_otp_codes`, `lib/email-otp.ts`), не Supabase Auth
- письма отправляются через Resend (`RESEND_API_KEY`, `EMAIL_FROM`)
- подробности — `docs/AUTH_MIGRATION.md`

### Telegram Mini App shell

Если используется `WebApp.requestFullscreen()`, нужно проверять поддержку метода и версию Telegram WebApp.  
Ошибка `WebAppMethodUnsupported` не должна ломать приложение.

## Важные правила работы

- не пересобирать архитектуру без необходимости
- вносить точечные изменения
- не ломать существующую логику, маршруты и интеграции
- переиспользовать текущие `features/`, `lib/`, `types/`
- если задача про UI, не трогать бизнес-логику без необходимости
- аккуратно работать с русским текстом и UTF-8
- web-версия не должна ломать Telegram Mini App
- Telegram Mini App auth и web auth должны сосуществовать
- admin API должен использовать существующую Telegram admin auth схему
- не ослаблять middleware ради новых admin endpoints

## Как лучше работать в новых task

Перед изменениями:

1. короткий анализ текущей реализации
2. краткий план
3. список файлов
4. preview diff
5. только после подтверждения применять правки

После изменений:

1. кратко описать, что изменено
2. перечислить файлы
3. дать QA-чеклист
4. явно указать, нужны ли SQL-миграции (Drizzle, `lib/db/migrations/`), новые env-переменные или настройки в BotFather

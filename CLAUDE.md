# Project Context

## Что это

`Don't worry club` — это Telegram Mini App для покерных турниров.

Проект уже развернут в новом окружении:

- новый GitHub репозиторий
- новый Vercel project
- новый Supabase project
- новый Telegram bot
- все env-переменные обновлены

Приложение уже рабочее:

- деплой есть
- база подключена
- схема перенесена
- Telegram Mini App открывается корректно

---

## Архитектура

- Next.js App Router
- TypeScript
- Tailwind CSS
- Supabase
- Telegram Mini App / Telegram Bot
- Vercel

---

## Основные сущности

- `players` — игроки, профиль, роль, Telegram-данные, ник
- `tournaments` — турниры, дата, лимит игроков, описание, локация, статус
- `registrations` — записи на турниры, waitlist, отмены
- `results` — результаты турниров, место, нокауты, очки рейтинга
- `seasons` — сезоны
- `player_achievements` — прогресс достижений игрока

---

## Структура проекта

- `app/` — страницы и UI (App Router)
- `app/api/` — API routes и webhook
- `features/` — бизнес-логика и работа с Supabase
- `lib/` — инфраструктура и интеграции
- `types/` — типы
- `components/` — UI-компоненты

---

## Что уже реализовано

- список турниров и страница турнира
- регистрация / waitlist / отмена
- профиль игрока
- рейтинг и результаты
- админка турниров
- Telegram webhook
- достижения
- server-side sync достижений

---

# Правила разработки

## Общие

- не пересобирать архитектуру без необходимости
- вносить минимальные и точечные изменения
- не ломать существующую логику
- не трогать несвязанные файлы
- переиспользовать `features`, `lib`, `types`
- не делать лишний рефакторинг
- аккуратно работать с UTF-8 и русским текстом

---

## Работа с задачами

- сначала найти минимальную точку изменения
- если можно решить в одном файле — решать в одном файле
- не менять структуру проекта без необходимости
- не усложнять решения

Перед изменениями:

- кратко опиши, что будешь менять

После изменений:

1. список изменённых файлов
2. кратко что сделано

Не нужно:

- длинный анализ
- лишние вопросы, если задача понятна
- дублирование контекста

---

## Особенности домена

- есть игроки с Telegram и могут быть manual игроки без Telegram
- `telegram_id` может быть `null`
- не вся логика должна зависеть от Telegram

- при удалении игрока из турнира:
  - удаляется только `registration`
  - `player` не удаляется

---

## Google Sheets

- используется синхронизация результатов турнира
- не ломать текущую интеграцию без необходимости
- UI не должен зависеть от ручного обновления из GS, если это можно исправить локально
- кнопка "Обновить из GS" — дополнительная, а не обязательная логика

---

## UI правила

- сохранять текущий стиль интерфейса
- не менять визуальную структуру без необходимости
- использовать существующие UI-паттерны
- изменения в админке должны быть простыми и понятными

---

## API и безопасность

- не утекать секреты
- использовать существующие API routes
- не дублировать серверную логику на клиенте

---

## Приоритет

1. корректность логики
2. минимальность изменений
3. стабильность UI
4. простота реализации

# Poker App: архитектурный план перехода с Supabase на PostgreSQL

> **АРХИВ — миграция завершена.** Все 10 Repository используют только
> PostgreSQL, Supabase-реализации и сам `DATABASE_PROVIDER`-переключатель
> удалены (не на что переключаться — Supabase-проект выведен из
> эксплуатации полностью, включая Auth и Storage). Раздел «Проблема границы
> Client Component ↔ Repository Layer» и связанное с ним архитектурное
> правило `"use server"` **остаются в силе** — это не история, а актуальная,
> действующая часть архитектуры (см. `docs/ADR-0001-server-actions-boundary.md`
> и README «Repository Layer»). Остальное содержимое документа — историческая
> запись плана, актуальному состоянию не соответствует.

Статус (на момент написания): **черновик на утверждение**. Это документ-анализ, не задание на реализацию. Ни кода, ни SQL, ни ORM-схемы, ни миграций в рамках подготовки этого документа не создавалось и не будет создаваться, пока план не согласован отдельно.

**Обновление.** В ходе реализации Этапа 3 (появление первой Postgres-реализации репозитория) обнаружена и устранена архитектурная блокирующая проблема, не связанная с PostgreSQL/Drizzle как таковыми — см. раздел «Проблема границы Client Component ↔ Repository Layer» ниже и `docs/ADR-0001-server-actions-boundary.md`. Proof of Concept подтверждён практическим экспериментом (сборка + проверка содержимого клиентского бандла + тесты). Это снимает архитектурное препятствие для продолжения Этапа 3, но не меняет статус документа в целом — остальные пункты Definition of Done (аудит, схема, backfill, паритет) по-прежнему требуют отдельного прохождения.

Этот документ детализирует Этапы 2–4 `docs/MIGRATION_PLAN.md` («Подготовка PostgreSQL» → «Поддержка двух провайдеров данных» → «Готовность к переключению»), которые в основном плане описаны кратко со ссылкой на методологию ReRaise. Здесь — тот же материал, но применительно конкретно к текущему состоянию Poker App после завершения Repository Layer, с более подробным разбором вопросов, специфичных именно для этого проекта.

Этапы 0–9 самой инфраструктурной/VPS-миграции (`MIGRATION_PLAN.md`) этим документом не отменяются и не дублируются — Docker, CI/CD, перенос Storage, переключение домена остаются там же, где были.

## Goal

Заменить Supabase как хранилище данных на PostgreSQL, сохранив бизнес-логику, API и наблюдаемое поведение приложения без изменений.

Конкретно: для каждого из 10 уже существующих Repository (`AppSettings`, `Activity`, `AvatarStorage`, `Season`, `Achievement`, `Player`, `Registration`, `TournamentLiveState`, `Result`, `Tournament`) появляется Postgres-реализация того же интерфейса, и вводится управляемый способ переключения между Supabase- и Postgres-реализацией без переписывания вызывающего кода (`features/*.ts`, `app/api/**`).

Repository Layer, завершённый на предыдущем этапе, — это ровно та граница, которая делает эту замену возможной без риска: весь доступ к данным уже сосредоточен в 10 файлах `Supabase*Repository.ts`, и ни один Feature/route не обращается к Supabase напрямую (кроме сознательно исключённого `middleware.ts`).

## Non-goals

Явно не входит в этот этап:

- **EmailAuth** — зависимость email-входа от `supabase.auth.*` (`app/api/auth/email-link/prepare/route.ts`, `app/page.tsx`, `app/login/page.tsx`) не решается здесь. Это отдельная, ранее зафиксированная как открытая, архитектурная проблема (см. врезку "Отличие от ReRaise, известное заранее" в `MIGRATION_PLAN.md`). Пока она не решена, Supabase не может быть отключён полностью — но сам переход БД на Postgres от неё не зависит и может идти параллельно.
- **Изменение бизнес-логики.** Ни правила турниров, ни модель рейтинга, ни waitlist-логика не меняются. Repository-методы остаются тонкими — фильтры/сортировки/join'ы, которые сегодня делает Supabase-запрос, переносятся 1:1.
- **Оптимизации.** Postgres-реализация не должна быть "лучше" Supabase-реализации по производительности как цель сама по себе — только эквивалентна по поведению. Если в ходе аудита схемы (см. ниже) найдутся реальные проблемы (дублирующиеся индексы, отсутствующие FK и т.п. — у ReRaise таких было несколько) — они фиксируются как находки, но их исправление не блокирует эту миграцию и не обязано происходить в её рамках.
- **Рефакторинг Feature.** `features/*.ts` не трогаются, кроме случаев, где сам факт появления двух реализаций Repository требует правки (таких случаев быть не должно — это и есть проверка того, что Repository Layer действительно герметичен).
- **Изменение API.** HTTP-статусы, форма ответов, сигнатуры экспортируемых функций — идентичны до и после.
- **VPS, Docker, CI/CD, перенос Storage, переключение домена.** Это отдельные, более поздние этапы (`MIGRATION_PLAN.md`, Этапы 5–9), которые не пересматриваются этим документом.
- **Удаление Supabase-реализаций и самого Supabase-проекта.** Не является частью Definition of Done этого этапа (см. ниже) — это сознательно отложенное, отдельное решение после burn-in периода.

## Current architecture

После завершения Repository Layer состояние проекта:

```
Route / Server Component
        ↓
     Feature            (features/*.ts — бизнес-логика, без изменений)
        ↓
   Repository           (lib/repositories/<domain>/{Interface,Supabase<Domain>Repository,index}.ts)
        ↓
   @/lib/supabase, @/lib/supabase-admin   (прямой импорт клиента — тот же паттерн сохраняется для Postgres, см. ниже)
        ↓
      Supabase
```

**10 репозиториев, все — тонкие обёртки, без общего базового класса:**

| Repository | Методов | Возвращает |
|---|---|---|
| `AppSettingsRepository` | 2 | скаляры (`boolean`) |
| `ActivityRepository` | 5 | domain rows + агрегаты |
| `AvatarStorageRepository` | 1 | URL (Storage, не таблица) |
| `SeasonRepository` | 9 | domain model (`SeasonRow`) |
| `AchievementRepository` | 3 | domain model |
| `PlayerRepository` | 30 | domain model (`Player`) + несколько DTO-проекций |
| `RegistrationRepository` | 18 | domain model + 6 JOIN-методов (raw rows) |
| `TournamentLiveStateRepository` | 5 | domain rows + 1 JOIN-метод (raw rows) |
| `ResultRepository` | 12 | domain model + rows + DTO |
| `TournamentRepository` | 14 | domain model (`Tournament`), без JOIN |

**Отличие от того, что предполагал Этап 0 `MIGRATION_PLAN.md`:** там упоминался "Database Client Layer" как отдельный, уже выделенный слой между Repository и Supabase (по аналогии с `lib/database/supabase/{server,browser,index}.ts` у ReRaise). По факту этот слой **не был построен** — каждый `Supabase*Repository.ts` импортирует `@/lib/supabase` (публичный клиент) или `@/lib/supabase-admin` (сервисная роль) напрямую. Это не было ошибкой предыдущего этапа — конструирование клиента у Poker App тривиально, выносить его в отдельный слой ради двух файлов не требовалось.

**Решено по итогам архитектурного ревью: отдельный Database Client Layer не вводится и для Postgres-стороны.** Вместо промежуточного слоя — один общий singleton-файл `lib/db/index.ts`, создающий и экспортирующий экземпляр Drizzle-клиента. Каждая `Postgres*Repository` импортирует этот singleton напрямую, тем же способом, каким `Supabase*Repository` сегодня импортирует `@/lib/supabase`. Дополнительной архитектурной прослойки между Repository и ORM не будет — см. Repository replacement strategy.

**Registration и TournamentLiveState содержат JOIN-методы**, возвращающие `Promise<any[]>` — сырые строки нескольких таблиц, сознательно нетипизированные (решение предыдущего этапа: агрегация/сборка бизнес-представления остаётся в Feature, а не в Repository). Это прямое требование к Postgres-реализации этих методов на Drizzle: в отличие от Supabase `select("*, players(*)")`, где результат — просто вложенный объект, Drizzle-запрос с JOIN нужно будет явно собрать в ту же форму (те же ключи, та же вложенность), не более строгую и не другую — форму определяет то, что уже потребляет Feature-слой, а не удобство ORM.

**Ни одна Postgres-инфраструктура ещё не существует** — ни контейнера, ни схемы, ни строки конфигурации. Аудит текущей Supabase-схемы (типы, PK/FK, индексы, RLS, мёртвые колонки) для Poker App **тоже не проводился** — в отличие от ReRaise, у которого перед этим этапом был отдельный read-only аудит (`POSTGRES_MIGRATION_AUDIT.md`). Это первое, что должно появиться в рамках Migration strategy ниже.

## Migration strategy

Порядок ниже отличается от списка в задании: добавлен аудит перед проектированием схемы, ORM зафиксирован решением (не отдельным шагом выбора), после Backfill добавлена отдельная Validation, а удаление Supabase-реализаций разведено с переключением на два отдельных шага. Каждое отличие аргументировано по ходу.

1. **Read-only аудит текущей Supabase-схемы Poker App.** Должен идти первым: без него шаг "Схема БД" — гадание, а не перенос. По методологии ReRaise (`POSTGRES_MIGRATION_AUDIT.md`): типы данных, PK/FK, индексы (в т.ч. дубликаты — у ReRaise нашлось 6 таких пар), RLS-политики, денормализация, мёртвые колонки. У Poker App 9 таблиц — заведомо меньше, чем 11 у ReRaise, аудит должен быть соразмерно короче.
2. **`lib/db/index.ts`** — singleton-файл, создающий и экспортирующий экземпляр Drizzle-клиента (решение зафиксировано, см. Repository replacement strategy). Не отдельный архитектурный слой — просто общая точка конструирования клиента, по аналогии с тем, как сегодня `@/lib/supabase` служит той же ролью для Supabase-стороны.
3. **Установка и настройка Drizzle ORM.** Выбор ORM больше не открытый вопрос (см. ниже) — на этом шаге только техническая настройка (зависимости, конфиг `drizzle.config.ts`, `drizzle-kit`).
4. **Инфраструктура PostgreSQL** — где физически живёт база (см. Database provider ниже; в `MIGRATION_PLAN.md` Этап 2 уже зафиксирован предварительный вариант — переиспользовать существующий контейнер `poker-clock-db` на VPS, отдельная БД + пользователь).
5. **Схема БД.**
6. **Миграции.**
7. **Backfill** — перенос существующих данных, идемпотентно, с сохранением исходных `uuid`.
8. **Validation.** Отдельный шаг сразу после Backfill, до начала работы над Postgres-реализациями Repository — проверка, что перенесённые данные корректны, прежде чем на них начнут писаться Repository-реализации. Подробности — см. новый одноимённый раздел ниже.
9. **Postgres-реализации всех 10 Repository** — аддитивная работа, ни один вызывающий код не меняется (см. Repository replacement strategy).
10. **Composition root + `DATABASE_PROVIDER`** — единая точка решения о провайдере (см. Repository replacement strategy и Database provider).
11. **Тестирование паритета** Supabase ⇄ Postgres на одинаковых входных данных.

Дальше — **не** "удаление Supabase-репозиториев и очистка" сразу за тестированием, а:

12. **Готовность подтверждена, переключение в проде — отдельное осознанное решение**, с burn-in периодом, в течение которого Supabase-реализации не удаляются (остаются путём отката).
13. **Удаление Supabase-реализаций и очистка** — только после burn-in, отдельным, письменно зафиксированным решением.

**Почему шаги 12–13 разделены, а не идут одним пунктом "удаление и очистка" сразу после тестирования:** это прямое повторение принципа, уже принятого в `MIGRATION_PLAN.md` (Этап 3 vs Этап 4) — «готовность к переключению» и «факт переключения» осознанно разведены, чтобы прохождение тестов паритета не создавало давления немедленно резать путь отката. У ReRaise это тоже отдельно зафиксировано как урок (`POSTGRES_MIGRATION_AUDIT.md`, п.13.9: «Poker Clock не сделал этого заранее и считает это упущением»). Удаление `Supabase*Repository.ts` файлов — необратимый шаг (в отличие от флага `DATABASE_PROVIDER`, который откатывается мгновенно), поэтому он не может быть частью Definition of Done одного и того же этапа, что и «переключатель готов и работает».

### Validation (после Backfill)

Отдельный, самостоятельный шаг между Backfill и началом работы над Postgres-реализациями Repository — данные должны быть подтверждены корректными прежде, чем на них будет опираться что-либо дальше. Включает:

- **Количество строк по каждой таблице** — точное совпадение `COUNT(*)` между Supabase и Postgres по каждому из 9 (10 с учётом Storage) доменов.
- **Выборочные проверки содержимого** — построчное сравнение случайной выборки записей (не только количества) по каждой таблице, включая денормализованные/вычисляемые поля (например, `results.rating_points`, перенесённое как есть, не пересчитанное).
- **Проверка ссылочной целостности** — все FK-связи (включая обнаруженные аудитом отсутствующие/добавленные FK, если такие решения были приняты на шаге схемы) целы, отсутствуют висячие ссылки.
- **Контрольные агрегаты** — несколько содержательных агрегатов, значимых для бизнес-логики (например, сумма `rating_points` по игроку, количество активных регистраций по турниру), совпадающих между Supabase и Postgres.

Только после того, как Validation пройдена полностью, начинается работа над Postgres-реализациями Repository (шаг 9) — если Validation находит расхождения, они устраняются на уровне backfill-скрипта до перехода дальше, а не откладываются "разберёмся потом".

## Repository replacement strategy

Решение о провайдере централизовано в одном **composition root** — новом файле `lib/repositories/provider.ts` — а не читается независимо каждым из 10 доменов. Это единственное место, которое обращается к `process.env.DATABASE_PROVIDER`:

```ts
// lib/repositories/provider.ts — composition root, единственное место чтения DATABASE_PROVIDER
export const databaseProvider: "supabase" | "postgres" =
  process.env.DATABASE_PROVIDER === "postgres" ? "postgres" : "supabase";
```

Каждый доменный `lib/repositories/<domain>/index.ts` — как и сегодня, синглтон-экспорт (см. `lib/repositories/tournament/index.ts` и аналогичные у всех 10 доменов), но больше не читает `process.env` сам — он потребляет уже resolved-значение из composition root:

Сейчас:
```ts
// lib/repositories/tournament/index.ts
export const tournamentRepository: TournamentRepository = new SupabaseTournamentRepository();
```

После этого этапа:
```ts
// lib/repositories/tournament/index.ts — потребитель решения, не источник
import { databaseProvider } from "../provider";

export const tournamentRepository: TournamentRepository =
  databaseProvider === "postgres"
    ? new PostgresTournamentRepository()
    : new SupabaseTournamentRepository();
```

Ключевые свойства:

- **Один источник истины.** `databaseProvider` вычисляется один раз при первом импорте `provider.ts` и переиспользуется всеми 10 доменами — а не читается заново в каждом `index.ts` независимо. Это делает рассинхронизацию между доменами (один видит `postgres`, другой в то же время — `supabase`) структурно невозможной, а не просто нежелательной — см. Database provider и решение о запрете частичного переключения ниже.
- **Импорты Feature/routes не меняются.** `features/tournaments.ts` и все остальные вызывающие модули по-прежнему импортируют `tournamentRepository` из `@/lib/repositories/tournament`, как и сегодня. Composition root — деталь, целиком замкнутая внутри `lib/repositories/`, не затрагивающая ни один файл за её пределами.
- Ни один файл вне `lib/repositories/tournament/` не импортирует `SupabaseTournamentRepository`/`PostgresTournamentRepository` напрямую (это уже проверено для Supabase-стороны на предыдущем этапе, и остаётся верным для Postgres-стороны по построению).

Каждая `Postgres*Repository` реализует тот же `Interface.ts`, что и `Supabase*Repository` — интерфейсы, зафиксированные на предыдущем этапе, не меняются. Единственное исключение, которое стоит явно проговорить заранее: JOIN-методы (`RegistrationRepository.findExportParticipants` и т.п., `Promise<any[]>`) должны в Postgres-реализации возвращать строки той же формы (те же ключи, та же вложенность), что и сейчас возвращает Supabase — форма этих строк определяется тем, что уже потребляет Feature-слой, а не удобством конкретного ORM.

## Проблема границы Client Component ↔ Repository Layer (обнаружена и решена)

Статус: **решено**, подтверждено Proof of Concept. Раздел документирует не только итоговое решение, но и путь к нему (обнаружение → отклонённые варианты → решение → проверка), чтобы будущим авторам не пришлось заново открывать те же тупиковые ветки. Полная запись решения в формате ADR — `docs/ADR-0001-server-actions-boundary.md`.

### Обнаруженная проблема

При добавлении первой Postgres-реализации репозитория (`PostgresPlayerRepository`, домен `player`) `next build` стал падать. **Причина не в PostgreSQL и не в Drizzle** — она в структуре импортов, существовавшей в проекте ещё до начала Postgres-миграции:

1. Часть `"use client"`-компонентов импортирует функции-значения напрямую из `features/*.ts` (`features/auth.ts`, `features/tournaments.ts`, `features/achievements.ts`), а не только через `app/api/**`. Подтверждено по факту: `app/page.tsx`, `app/tournaments/page.tsx`, `app/tournaments/[id]/page.tsx`, `app/players/[id]/page.tsx`, `app/players/[id]/achievements/page.tsx`, `app/my-tournaments/page.tsx`, `app/leaderboard/page.tsx`, `app/admin/tournaments/page.tsx`, `app/admin/tournaments/[id]/edit/page.tsx`, `app/admin/tournament-notifications/page.tsx`, `app/admin/results/[id]/page.tsx` — 11 файлов.
2. `features/*.ts` были обычными TS-модулями без какой-либо границы Client/Server — никакой `"use server"` в них не было.
3. Каждый доменный `lib/repositories/<domain>/index.ts` делает **eager static import** обеих реализаций сразу (`SupabaseXxxRepository` и `PostgresXxxRepository`) и только потом ветвится по `databaseProvider` в рантайме (см. «Repository replacement strategy» выше) — то есть сам факт наличия файла `PostgresXxxRepository.ts` в дереве импортов уже втягивает его в граф сборки, независимо от значения флага.
4. Postgres-репозитории используют пакет `postgres` (`drizzle-orm/postgres-js`), зависящий от Node core-модулей `net`/`tls`, которых нет в браузерном окружении.

Вместе эти четыре факта означают: у Next.js не было ни единого сигнала о том, что `features/tournaments.ts` (и всё, что за ним) должно остаться на сервере. Граф `"use client"`-страница → `features/tournaments.ts` → `lib/repositories/tournament/index.ts` → `PostgresTournamentRepository.ts` → `postgres` целиком включался в клиентский бандл, и сборщик падал на резолве `net`/`tls` для браузерного окружения.

### Отклонённые варианты (и почему)

| Вариант | Результат | Почему отклонён |
|---|---|---|
| `import "server-only"` в каждом `Postgres*Repository.ts` | Сборка падает — но осмысленно: `server-only` явно называет нарушивший файл | Runtime/сборочная проверка постфактум, а не механизм, который *убирает* файл из клиентского графа. Полезен как defense-in-depth (и остаётся в проекте — см. правила ниже), но сам по себе проблему не решает: граф всё равно строится и падает. |
| `package.json` с полем `"browser"` + `index.browser.ts`/`index.server.ts` (POC на домене `player`) | Настоящий browser-бандл переставал резолвить `postgres` | Next.js App Router отдельно строит граф **Client Component SSR** (рендеринг клиентских компонентов на сервере) — этот граф резолвит `"main"`, а не `"browser"`, и продолжает утыкаться в `index.server.ts` → `postgres`. `next build` продолжал падать. К тому же поддержка `package.json#browser` пришлось бы масштабировать вручную на все 10 доменов — обходной путь, не архитектурное решение. |
| Изучение архитектуры ReRaise (родственный проект на той же инфраструктуре, уже прошедший этот же переход Supabase → Postgres/Drizzle) | Обнаружено: `features/tournaments.ts` и `features/auth.ts` в ReRaise начинаются с `"use server"` — а `features/achievements.ts`/`features/admin.ts` этой директивы не несут, потому что ни один Client Component ReRaise их напрямую не импортирует | Это и есть настоящий механизм границы — см. «Принятое решение» ниже. |
| PoC: `"use server"` в трёх Feature-файлах Poker App (`auth.ts`, `tournaments.ts`, `achievements.ts`) + полный откат POC-эксперимента с `browser`-полем | `next build` проходит и при `DATABASE_PROVIDER` не заданном (supabase), и при `DATABASE_PROVIDER=postgres`; `postgres`/`drizzle-orm` подтверждённо отсутствуют в `.next/static`, но присутствуют в `.next/server` | Подтверждено практическим экспериментом — см. «Проверка (Proof of Concept)» ниже. Принято как решение. |

### Принятое решение

**Feature-модули, которые напрямую импортируются из Client Components, должны начинаться с директивы `"use server"`.**

`"use server"` — не runtime-guard (как `server-only`), а **компиляторная граница React Server Components**: Next.js вырезает тела экспортированных функций (и весь хвост их импортов — Repository, Postgres-драйвер, `net`/`tls`) из клиентского графа ещё до того, как сборщик пытается резолвить модули. Клиентский компонент получает лёгкую ссылку на серверное действие (RPC-вызов), а не тело функции и не код репозитория.

Применено к (проверено, что именно эти файлы реально импортируются `"use client"`-компонентами — директива ставится по факту пересечения границы, не «на всякий случай»):
- `features/auth.ts`
- `features/tournaments.ts`
- `features/achievements.ts`

**Не** применено к `features/admin.ts`, `features/settings.ts` — ни один Client Component их напрямую не импортирует, необходимости в границе там нет. В этом Poker App отличается от ReRaise: у ReRaise клиентские страницы не дотягиваются до `features/achievements.ts` напрямую, поэтому там директива не понадобилась вовсе; у Poker App `app/players/[id]/achievements/page.tsx` и `app/players/[id]/page.tsx` импортируют `getPlayerAchievements` напрямую — поэтому `features/achievements.ts` тоже требует `"use server"`, хотя в ReRaise аналогичный файл её не несёт.

### Ограничение, найденное и устранённое

Файл с `"use server"` может экспортировать только async-функции и `export type` (типы стираются компилятором и не участвуют в рантайме Server Actions). `features/auth.ts` экспортировал константу `TERMS_VERSION` вперемешку с async-функциями — это конфликтует с ограничением. Решение — по образцу ReRaise: константа вынесена в отдельный модуль `lib/terms.ts`, `features/auth.ts` и `app/page.tsx` импортируют её оттуда. Других конфликтов такого рода в `tournaments.ts`/`achievements.ts` не найдено — все их экспорты уже были async-функциями или типами.

Также проверена сериализуемость аргументов/возвратов всех функций, ставших Server Actions: только примитивы, plain-объекты/массивы и `Map<string, number>` (последний нативно поддерживается протоколом сериализации React для Server Actions, не JSON-протоколом — это не проблема). Функций, class-инстансов или `Symbol` в сигнатурах не найдено.

### Проверка (Proof of Concept)

- `tsc --noEmit` — чисто (не считая предсуществующего, несвязанного с этой задачей пробела в типах одного vitest-теста).
- `next build` при `DATABASE_PROVIDER` не заданном (supabase, по умолчанию) — успешно.
- `next build` при `DATABASE_PROVIDER=postgres` — успешно, без деградации скорости сборки (ленивый Drizzle-прокси в `lib/db/index.ts` не активировался при сборке — реального соединения с БД не потребовалось ни разу).
- Поиск по `.next/static` (клиентский бандл) на `postgres-js`, `PostgresError`, `drizzle-orm`, `node:net`, `node:tls` — ни одного совпадения в обеих сборках.
- Те же строки **присутствуют** в `.next/server/**` — граница реально разделяет графы, а не «нигде ничего нет».
- `npx vitest run` — 22/22 теста, 5/5 файлов, без изменений в результатах.

Гипотеза подтверждена практическим экспериментом. Архитектурное препятствие для продолжения PostgreSQL-миграции устранено. Правила закреплены в разделе «Архитектурные правила» ниже и продублированы в `docs/MIGRATION_PLAN.md` (Этап 1, Этап 3).

## Архитектурные правила (закреплено после PoC)

### Feature Layer

- Если Feature-модуль (`features/*.ts`) импортируется хотя бы одним Client Component — файл должен начинаться с `"use server"`.
- Такой файл может экспортировать только async-функции и `export type`.
- Константы и любые другие non-function-значения, нужные и Feature-модулю, и клиентскому коду, выносятся в отдельный модуль без `"use server"` (пример: `lib/terms.ts`).
- Директива не добавляется «на всякий случай» туда, где ни один Client Component не импортирует файл напрямую — она должна отражать реальный факт пересечения клиентской границы (пример: `features/admin.ts`, `features/settings.ts` её не несут).

### Repository Layer

- Repository Layer не использует browser/server split (`index.browser.ts`/`index.server.ts`) и не использует `package.json#browser` как архитектурное решение — этот путь проверен и отклонён (см. выше): App Router строит отдельный граф Client Component SSR, резолвящий `"main"`, а не `"browser"`, поэтому расщепление не устраняет проблему полностью.
- Единственная точка ветвления — composition root `databaseProvider` (`lib/repositories/provider.ts`), потребляемый каждым доменным `index.ts` (см. «Repository replacement strategy» выше). Ни один доменный `index.ts`, включая `player`, не отклоняется от этого паттерна.

### PostgreSQL

- Postgres-репозитории остаются обычными серверными реализациями — никакой специальной сборочной обработки, кроме стандартного `import "server-only"`.
- `server-only` в каждом `Postgres*Repository.ts` — защитный (defense-in-depth) механизм на случай четвёртого пути импорта в обход `"use server"`-границы в будущем, а не основной способ разделения client/server. Основная граница — `"use server"` в Feature-слое.

## Database provider

`DATABASE_PROVIDER` — рантайм переменная окружения, читаемая **один раз**, в единственном месте — composition root `lib/repositories/provider.ts` (см. Repository replacement strategy выше). Ни один из 10 доменных `index.ts` не обращается к `process.env` самостоятельно.

Поддерживаются только два режима, оба — глобальные, на всё приложение сразу:

- `DATABASE_PROVIDER=postgres` — все 10 репозиториев используют Postgres-реализацию.
- не выставлено, или любое другое значение (включая явное `supabase`) — все 10 репозиториев используют Supabase-реализацию, то есть **сегодняшнее поведение**, без изменений.

**Частичное переключение по доменам не поддерживается — это принятое архитектурное решение, не открытый вопрос.** Не должно существовать состояния, когда часть репозиториев смотрит в Postgres, а часть — в Supabase одновременно: при перекрёстных зависимостях между доменами (Registration зависит от Player/Season/Tournament) такое расщепление могло бы дать наблюдаемо неверные ответы, даже если каждый Repository по отдельности корректен. Единая точка чтения флага (composition root) делает это состояние структурно невозможным, а не просто запрещённым по соглашению.

Значение по умолчанию — `supabase`, то есть добавление composition root и самого переключателя в кодовую базу **не меняет поведение прода**, пока флаг не выставлен явно. Это тот же принцип "постепенное внедрение рядом, а не вместо", что и во всём остальном плане миграции.

## Known limitations

Не риски миграции, а сознательные архитектурные исключения, действующие в рамках этого этапа независимо от его результата:

- **EmailAuth.** Email-вход зависит от `supabase.auth.*` напрямую (`app/api/auth/email-link/prepare/route.ts`, `app/page.tsx`, `app/login/page.tsx`). Эти данные (OTP-коды, пользователи Auth) живут в служебной схеме Supabase Auth, а не в одном из 10 репозиториев, участвующих в этом этапе. Переключение `DATABASE_PROVIDER=postgres` никак не затрагивает email-вход — он продолжает идти через Supabase Auth независимо от значения флага, пока EmailAuth-домен не будет решён отдельно (см. `MIGRATION_PLAN.md`, врезка "Отличие от ReRaise, известное заранее"). Практическое следствие: даже при `DATABASE_PROVIDER=postgres` Supabase как проект не может быть отключён полностью.
- **Realtime.** `supabase.channel(...).on("postgres_changes", ...)` (`app/page.tsx`, `app/tournaments/page.tsx`) — managed WS-слой поверх логической репликации Supabase, не имеющий прямого аналога в Postgres/Drizzle. Не переключается вместе с `DATABASE_PROVIDER` и остаётся на Supabase независимо от того, куда смотрят Repository. Не считается регрессией при тестировании паритета — это заранее согласованное исключение, а не нерешённая проблема.

## Risks

- **Схема ещё не audited.** Без read-only аудита неизвестно, есть ли у Poker App аналоги находок ReRaise — задвоенные индексы, отсутствующие FK (у ReRaise: `results.season_id` без FK), мёртвые колонки, RLS-политики без реального содержания. Решение о схеме "вслепую" рискует зафиксировать существующие проблемы в новой БД, а не исправить или хотя бы явно принять их.
- **Типы данных.** `telegram_id` (`bigint`) должен маппиться так, чтобы TS-тип остался `number`, как сейчас (`PlayerRow.telegram_id: number`) — у ReRaise это конкретно означало `bigint({ mode: 'number' })`, а не `{ mode: 'bigint' }`; для Poker App нужна та же проверка, не предположение по аналогии.
- **JOIN-методы (Registration, TournamentLiveState).** Возвращают нетипизированные `any[]` сознательно — риск в том, что Postgres-реализация этих 7 методов (6 в Registration + 1 в TournamentLiveState) может незаметно вернуть данные в чуть другой форме (порядок ключей не важен, но вложенность/naming — важны, т.к. Feature обращается к конкретным полям).
- **Backfill на живых данных при непрерывной записи.** В отличие от статичного дампа, `results`, `registrations`, `tournament_live_entries` могут писаться во время живого турнира. Разовый backfill перед переключением рискует не подхватить записи, сделанные между моментом backfill и моментом реального флипа `DATABASE_PROVIDER` — нужен явно решённый протокол финального окна (см. Open Questions), а не предположение, что один прогон достаточен. Шаг Validation (см. Migration strategy) снижает, но не устраняет этот риск — он проверяет корректность уже перенесённых данных, а не покрывает записи, случившиеся после переноса.
- **`drizzle-kit migrate` CLI имеет известный баг** (задокументированный ReRaise — молчаливое проглатывание ошибок применения миграций) — потребуется тот же обходной путь, что и у ReRaise: собственный скрипт поверх `drizzle-orm/postgres-js/migrator`, не полагаться на штатный CLI.
- **Переменная окружения не доходит до контейнера/процесса.** У ReRaise конкретно `docker-compose.yml` не прокидывал `DATABASE_PROVIDER` внутрь контейнера — ошибка обнаружилась не сразу. Для Poker App это пока не актуально (прод ещё на Vercel), но проверка `printenv` перед реальным переключением обязательна, когда дойдёт до VPS-этапа.
- **Composition root — единая точка отказа.** Централизация решения о провайдере в одном `provider.ts` (см. Repository replacement strategy) устраняет риск рассинхронизации между доменами, но взамен даёт более широкий радиус поражения у любой ошибки в этом одном файле — она сразу затрагивает все 10 доменов одновременно, а не один изолированный. Митигируется тем, что сам файл предельно простой (одна строка логики) и полностью покрывается тестированием паритета перед любым реальным переключением.
- **Производительность под нагрузкой** — паритет по поведению не проверяет паритет по нагрузке; реальная разница (PostgREST vs прямой доступ) проявится только на проде в момент турнира — то же ограничение, что уже зафиксировано в `MIGRATION_PLAN.md` (Этап 4, риски).

## Definition of Done

Этот этап считается завершённым, когда:

- [ ] Read-only аудит текущей Supabase-схемы Poker App проведён и задокументирован (по методологии, аналогичной `POSTGRES_MIGRATION_AUDIT.md` ReRaise).
- [ ] `lib/db/index.ts` создан — общий singleton, экспортирующий экземпляр Drizzle-клиента.
- [ ] Drizzle ORM установлен и настроен (`drizzle.config.ts`, `drizzle-kit`).
- [ ] PostgreSQL развёрнут (используется существующий контейнер `poker-clock-db` либо иное согласованное решение), изолирован от базы ReRaise на уровне СУБД.
- [ ] Схема спроектирована и применена через миграции.
- [ ] Backfill проведён (паритет количества строк по каждой из 10 таблиц/доменов), включая явное решение по протоколу финального окна синхронизации перед реальным переключением.
- [ ] Validation после Backfill пройдена полностью (количество строк, выборочные проверки содержимого, ссылочная целостность, контрольные агрегаты).
- [ ] Postgres-реализации всех 10 Repository существуют, включая корректное воспроизведение формы JOIN-методов (`any[]`) там, где они есть.
- [x] Composition root (`lib/repositories/provider.ts` или аналог) — единственная точка чтения `DATABASE_PROVIDER`; ни один доменный `index.ts` не читает `process.env` самостоятельно. Значение по умолчанию (`supabase`) не меняет текущее поведение прода.
- [x] Проблема границы Client Component ↔ Repository Layer устранена: Feature-модули, импортируемые Client Components, помечены `"use server"`; подтверждено PoC — `next build` проходит при `DATABASE_PROVIDER=postgres`, `postgres`/`drizzle-orm` отсутствуют в `.next/static`. См. раздел «Проблема границы Client Component ↔ Repository Layer» выше и `docs/ADR-0001-server-actions-boundary.md`.
- [ ] Проверка паритета Supabase ⇄ Postgres пройдена на одинаковых входных данных для всех 10 доменов.
- [ ] Приложение по-прежнему полностью работает через Supabase в проде — ничего из этого этапа не обязывает переключаться немедленно.

**Явно НЕ входит в Definition of Done этого этапа** (в отличие от предложенного в задании списка): фактическое переключение прода на `DATABASE_PROVIDER=postgres`, прохождение burn-in периода, удаление Supabase-реализаций, удаление самого Supabase-проекта. Эти пункты — предмет отдельного решения после того, как этот этап завершён (соответствует Этапам 4+ `MIGRATION_PLAN.md`).

## Open questions

Архитектурные решения по ORM, Database Client Layer, централизации `DATABASE_PROVIDER` и запрету частичного переключения по доменам приняты (см. соответствующие разделы выше) и больше не значатся здесь как открытые. Остаются только вопросы, у которых пока действительно нет решения:

1. **Протокол backfill при непрерывной записи.** Один разовый прогон перед переключением, или требуется механизм финальной досинхронизации (короткое окно между последним backfill и флипом флага, во время которого запись в "старую" БД либо приостанавливается, либо дублируется)? Особенно значимо для `tournament_live_entries`/`results`, активно пишущихся во время турнирных дней.
2. **Критерии и длительность burn-in периода** после фактического переключения (Этап 4 `MIGRATION_PLAN.md`) — не определяются этим документом, но стоит зафиксировать явно, что решение принимается отдельно, ближе к моменту, а не сейчас задним числом.
3. **Где физически живёт Postgres, если к моменту начала реализации это ещё не подтверждено.** `MIGRATION_PLAN.md` предварительно фиксирует переиспользование контейнера `poker-clock-db` (отдельная БД + пользователь). Если на момент начала этого этапа появятся новые обстоятельства (например, нагрузка ReRaise на этот же контейнер) — решение нужно будет подтвердить или пересмотреть явно, а не по умолчанию.
